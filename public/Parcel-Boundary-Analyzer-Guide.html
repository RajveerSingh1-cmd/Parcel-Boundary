1. What Does This App Do?

The Parcel Boundary Analyzer is a web application designed for urban planners. Here's what it does in simple terms:

  1. Upload — You upload a text file containing geographic coordinates (latitude & longitude points)
  2. Parse — The app reads the file, extracts valid coordinates, and skips bad ones
  3. Analyze — It finds the two closest points using a smart algorithm
  4. Visualize — It shows everything on an interactive map

💡 Real-World Use Case

Imagine you're an urban planner with hundreds of boundary points for land parcels. You need to find which two boundary markers are closest together — maybe to check for overlap or potential conflicts. This app does that automatically!

What is a "Parcel"?

In urban planning, a parcel is a piece of land. Each parcel has boundary points — the corners that define its shape. When you connect these points, you get the parcel boundary (a polygon shape on the map).

4. How the App Works — Data Flow

When you upload a file, here's exactly what happens, step by step:

📁 User drops file → Upload.jsx reads it → App.jsx receives text → parser.js extracts points → closestPair.js finds pair → 🗺️ Map shows results

Detailed Flow:

  1. User drags a .txt file onto the upload zone
  2. Upload.jsx uses the browser's FileReader API to read the file as text
  3. App.jsx receives the text via a callback function
  4. App checks the file size:
    • Small file (< 5,000 lines) → processes directly on the main thread
    • Large file (≥ 5,000 lines) → sends to a Web Worker (background thread)
  5. parser.js reads each line, extracts lat/lon/id, and skips bad lines
  6. closestPair.js finds the two closest points:
    • Under 1,000 points → uses brute force (compare every pair)
    • 1,000+ points → uses divide & conquer (much faster)
  7. distance.js calculates real-world distance using the Haversine formula
  8. Results are stored in React state → triggers a re-render
  9. MapView.jsx plots markers, polygons, and the closest pair line
  10. ResultsPanel.jsx shows the numbers in the sidebar

7a. distance.js — The Haversine Formula

Problem: Earth is a sphere, not a flat surface. You can't just use the Pythagorean theorem (a² + b² = c²) to find the distance between two GPS coordinates. You need a formula that accounts for the curve of the Earth.

Solution: The Haversine formula calculates the shortest distance between two points on a sphere (called the "great-circle distance").

src/utils/distance.js
export function haversineDistance(lat1, lon1, lat2, lon2) {
  const R = 6371000;  // Earth's radius in meters (≈ 6,371 km)

  // Step 1: Convert degrees to radians (math requires radians)
  const toRad = (deg) => (deg * Math.PI) / 180;

  // Step 2: Find the differences in latitude and longitude
  const dLat = toRad(lat2 - lat1);   // Δ latitude in radians
  const dLon = toRad(lon2 - lon1);   // Δ longitude in radians

  // Step 3: Apply the Haversine formula
  const a =
    Math.sin(dLat / 2) * Math.sin(dLat / 2) +           // sin²(Δlat/2)
    Math.cos(toRad(lat1)) * Math.cos(toRad(lat2)) *   // cos(lat1)·cos(lat2)
    Math.sin(dLon / 2) * Math.sin(dLon / 2);             // · sin²(Δlon/2)

  // Step 4: Calculate angular distance
  const c = 2 * Math.atan2(Math.sqrt(a), Math.sqrt(1 - a));

  // Step 5: Multiply by Earth's radius to get meters
  return R * c;
}

💡 Haversine in Plain English

Imagine stretching a string between two cities on a globe. The Haversine formula calculates the length of that string. It gives you the shortest path along Earth's surface (not a straight tunnel through the Earth).


Example: New York (40.7128, -74.0060) to the Statue of Liberty (40.6892, -74.0445) ≈ 2,836 meters.

export function parseCoordinates(text) {
  // Split text into lines, remove empty lines
  const lines = text.split(/\r?\n/).filter(l => l.trim() !== '');
  const points = [];     // Good coordinates go here
  const skipped = [];    // Bad lines go here (with reason)

  // Detect header: does the first line have letters (like "lat,lon")?
  const firstLine = lines[0].trim();
  if (/[a-df-zA-DF-Z]/.test(firstLine)) {
    hasHeader = true;     // Skip this line
  }

  // Process each line
  for (let i = startIdx; i < lines.length; i++) {
    try {
      const parsed = parseLine(line, i + 1);  // Try to parse it
      points.push(parsed);                    // Success! Add to good list
    } catch (err) {
      skipped.push({ line: i+1, reason: err.message }); // Bad line, record why
    }
  }

  return { points, skipped, totalLines };
}

// Parse ONE line → { lat, lon, id }
function parseLine(line) {
  // Split by comma or space
  let parts = line.includes(',')
    ? line.split(',')       // "40.71,-74.00" → ["40.71", "-74.00"]
    : line.split(/\s+/);    // "40.71 -74.00" → ["40.71", "-74.00"]

  const lat = parseFloat(parts[0]);   // Convert string → number
  const lon = parseFloat(parts[1]);
  const id  = parts[2] || null;       // Optional parcel ID

  // Validate: latitude must be -90 to 90, longitude -180 to 180
  if (lat < -90 || lat > 90) throw new Error('Latitude out of range');
  if (lon < -180 || lon > 180) throw new Error('Longitude out of range');

  return { lat, lon, id };
}

🟢 Why use try/catch?

If one line is bad (like "hello,world"), we don't want the WHOLE file to fail. The try/catch pattern catches the error for that line, records it, and moves on to the next line. This is called graceful error handling.

7c. closestPair.js — The Core Algorithm

This file contains two algorithms to find the two closest points:

Algorithm 1: Brute Force (for < 1,000 points)

Idea: Compare EVERY pair of points. Simple but slow for large datasets.

// Compare every pair: (A,B), (A,C), (A,D), (B,C), (B,D), (C,D), ...
for (i = 0; i < points.length; i++) {
  for (j = i + 1; j < points.length; j++) {
    distance = haversineDistance(points[i], points[j]);
    if (distance < minimum) {
      minimum = distance;   // Found a closer pair!
    }
  }
}

⚠️ Performance

Brute force with 1,000 points = 499,500 distance calculations
Brute force with 10,000 points = 49,995,000 calculations ← TOO SLOW!
That's why we use a faster algorithm for large datasets.

Algorithm 2: Divide and Conquer (for ≥ 1,000 points)

Idea: Split the problem in half, solve each half, then check the boundary.

// Simplified steps:

// 1. Sort all points by latitude (left to right)
// 2. Split into LEFT half and RIGHT half
// 3. Find closest pair in LEFT half  (recursion!)
// 4. Find closest pair in RIGHT half (recursion!)
// 5. Take the smaller of those two distances → call it δ (delta)
// 6. Check the "strip" near the middle line
//    (closest pair might span left-right, but must be within δ)
// 7. Return the overall closest pair
LEFT HALF │ RIGHT HALF │ • • │ • • │ • • • • │ • │ ←── Find closest ──→ │ ←── Find closest ──→ │ Then check the "strip" around the middle line ◄──── δ ────►│◄──── δ ────►

💡 Why is Divide and Conquer faster?

Brute force: O(n²) — 10,000 points = ~50 million comparisons
Divide & Conquer: O(n log n) — 10,000 points = ~130,000 comparisons

That's 380x fewer calculations! The "trick" is that by splitting the problem, we eliminate most pairs from consideration.

8. Web Worker — Background Processing

The Problem

JavaScript runs on a single thread by default. If you do heavy math (like computing distances for 10,000 points), the browser freezes — buttons stop working, animations freeze, the page becomes unresponsive.

The Solution: Web Workers

A Web Worker runs JavaScript in a separate background thread. The main thread (UI) stays responsive while the worker does the heavy lifting.

┌─────────────────────┐ ┌─────────────────────┐ │ MAIN THREAD │ │ WEB WORKER │ │ (UI stays fast) │ │ (background) │ │ │ │ │ │ User clicks button │────────►│ Parse 10,000 lines │ │ Spinner shown │ │ Find closest pair │ │ Map still works │ │ Calculate distance │ │ │◄────────│ Send back results │ │ Display results │ │ │ └─────────────────────┘ └─────────────────────┘ postMessage() ←──── Communication ────→ postMessage()
src/workers/compute.worker.js
import { parseCoordinates } from '../utils/parser.js';
import { findClosestPair } from '../utils/closestPair.js';

// Listen for messages from the main thread
self.onmessage = function(e) {
  const { type, payload } = e.data;

  if (type === 'parse') {
    // Parse the file text
    const result = parseCoordinates(payload.text);
    self.postMessage({ type: 'parseResult', data: result });
    // ↑ Send results back to main thread
  }

  if (type === 'compute') {
    // Find the closest pair
    const start = performance.now();
    const result = findClosestPair(payload.points);
    const elapsed = performance.now() - start;
    self.postMessage({
      type: 'computeResult',
      data: { ...result, computeTime: elapsed }
    });
  }
};

💡 How Workers Communicate

Workers can't touch the DOM (no document.getElementById). They can only send and receive messages using postMessage() and onmessage. Think of it like passing notes — the worker sends a note with the answer, and the main thread reads it.

onDrop = (e) => {

};

11. Key React Concepts Explained

What is a Component?

A component is a reusable piece of UI. Each component is a JavaScript function that returns JSX (HTML-like code).

// A simple component
function Greeting({ name }) {
  return <h1>Hello, {name}!</h1>;
}

// Using it
<Greeting name="Farhan" />    // Renders: <h1>Hello, Farhan!</h1>

What is State? (useState)

"State" is data that can change over time. When state changes, React re-renders the component.

const [count, setCount] = useState(0);  // count starts at 0

// When you call setCount(5), React:
//   1. Updates count to 5
//   2. Re-renders the component with the new value
//   3. The screen updates automatically!

What is useEffect?

Runs code after the component renders. Used for side effects like fetching data, setting up listeners, or changing the DOM.

useEffect(() => {
  document.title = 'New Title';   // Runs after render
}, [someVariable]);                   // Only re-runs when someVariable changes

What is useCallback?

Prevents functions from being re-created on every render. Important for performance.

// Without useCallback: new function created every render
const handleClick = () => { ... };

// With useCallback: same function reused unless dependencies change
const handleClick = useCallback(() => { ... }, []);

What is useRef?

Holds a reference to a DOM element or a value that persists across renders without causing re-renders.

const inputRef = useRef(null);

// Later, you can access the DOM element:
inputRef.current.click();   // Programmatically click the input

What are Props?

Props are data passed from parent to child. Like function arguments!

// Parent passes data
<ResultsPanel closestPair={pair} computeTime={42} />

// Child receives it
function ResultsPanel({ closestPair, computeTime }) {
  return <p>Time: {computeTime}ms</p>;
}
